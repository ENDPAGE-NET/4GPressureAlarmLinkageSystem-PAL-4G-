# PAL_4G 部署文档

## 1. 端口清单

| 服务 | 默认端口 | 是否必须公网开放 | 说明 |
| --- | --- | --- | --- |
| Web 前端 | `5173` | 否 | 本地开发端口，正式环境通常走 `80/443` 反向代理 |
| 后端 API | `8001` | 否 | 本地开发端口，正式环境通常走 `80/443` 反向代理到后端 |
| MQTT Broker 明文 | `1883` | 可选 | 仅联调用，正式环境不建议直接暴露 |
| MQTT Broker TLS | `8883` | 是 | 真机直连建议正式使用该端口 |
| HTTPS | `443` | 是 | Web、API、证书校验、后续管理入口 |
| HTTP | `80` | 可选 | 可仅用于跳转到 `443` |

说明：

- 正式环境不建议让用户直接访问 `5173` 或 `8001`
- 正式环境建议统一通过域名和 `443`
- 真机 MQTT 直连优先使用 `8883`

## 3. 服务器部署还需要什么

| 类别 | 是否必须 | 需要内容 | 说明 |
| --- | --- | --- | --- |
| 公网服务器 | 必须 | 1 台 Linux 服务器或云主机 | 用于部署 Web、后端、MQTT Broker |
| 域名 | 必须 | 网站域名和 MQTT 子域名 | 例如 `usr-iot.endpage.net`、`mqtt.usr-iot.endpage.net` |
| DNS 解析 | 必须 | 域名解析到服务器公网 IP | Web 和 MQTT 都要可访问 |
| HTTPS 证书 | 必须 | 网站证书 | 给 `443` 用 |
| MQTT TLS 证书 | 必须 | Broker 证书链 | 真机若走 TLS，需要 Broker 证书可校验 |
| MQTT Broker | 必须 | 推荐 `EMQX` | 负责真机和后端的 MQTT 消息中转 |
| 反向代理 | 建议 | Nginx 或同类 | 统一对外暴露 `443` |
| 数据库备份策略 | 建议 | SQLite 备份或迁移方案 | 交付前应确定备份方式 |
| 防火墙规则 | 必须 | 放行 `443`、`8883` | 必要时临时放行 `1883` 联调 |

## 4. 后端部署到服务器前还需要填写的配置

需要在服务器上的 `backend/.env` 中补齐或确认这些配置：

| 配置项 | 是否必须 | 当前状态 | 说明 |
| --- | --- | --- | --- |
| `SECRET_KEY` | 必须 | 需替换 | 不能继续用默认值 |
| `DATABASE_URL` | 必须 | 已有默认值 | 可先继续用 SQLite，本地文件路径需确认 |
| `MQTT_ENABLED` | 必须 | 当前默认 `false` | 正式联调要改为 `true` |
| `MQTT_BROKER_HOST` | 必须 | 未确定 | 填真实 Broker 域名或 IP |
| `MQTT_BROKER_PORT` | 必须 | 未确定 | 建议正式使用 `8883` |
| `MQTT_USERNAME` | 必须 | 未确定 | 由 Broker 分配 |
| `MQTT_PASSWORD` | 必须 | 未确定 | 由 Broker 分配 |
| `MQTT_CLIENT_ID` | 必须 | 已有默认值 | 后端连接 Broker 的客户端 ID |
| `MQTT_TLS_ENABLED` | 必须 | 已支持 | TLS 联调时改为 `true` |
| `MQTT_TLS_CA_CERTS` | 建议 | 未提供 | CA 证书路径 |
| `MQTT_TLS_CERTFILE` | 可选 | 未提供 | 双向 TLS 时才需要 |
| `MQTT_TLS_KEYFILE` | 可选 | 未提供 | 双向 TLS 时才需要 |
| `MQTT_TLS_INSECURE` | 必须确认 | 默认 `false` | 正式环境不建议改成 `true` |

## 5. 真机接入联调还需要客户提供什么

| 类别 | 是否必须 | 需要客户提供的内容 | 说明 |
| --- | --- | --- | --- |
| 真机 MQTT 参数修改权限 | 必须 | 能修改设备 MQTT 配置 | 否则设备无法切到你们自己的 Broker |
| 真机真实 SN | 必须 | 每台设备的 SN | 后端设备建档和 topic 识别依赖它 |
| 真机真实 IMEI | 建议 | 每台设备的 IMEI | 便于排查和辅助识别 |
| 状态上报样例 | 必须 | 一条真实 `status` payload | 用于核对字段映射 |
| 报警上报样例 | 必须 | 一条真实 `alarm` payload | 用于核对报警字段和联动触发 |
| 执行反馈样例 | 必须 | 一条真实 `feedback` payload | 用于核对命令回执处理 |
| 断开命令格式 | 必须 | 真机收到什么表示“断开” | 如果不是当前 JSON，需要改后端 |
| 恢复命令格式 | 必须 | 真机收到什么表示“恢复” | 如果不是当前 JSON，需要改后端 |
| TLS 兼容性 | 必须确认 | 真机是否支持校验证书 | 决定最终 Broker TLS 配置方式 |
| 现场网络 | 必须确认 | SIM 卡能否访问公网 Broker | 不通则无法联调 |

## 6. 当前后端与真机对接前的差距

| 项目 | 当前是否完成 | 还差什么 |
| --- | --- | --- |
| 后端 MQTT 代码 | 是 | 无阻塞 |
| 后端 TLS 能力 | 是 | 需要真实证书路径和 Broker 地址 |
| MQTT Broker | 否 | 需要部署 |
| 真机改连自有 Broker | 否 | 需要现场修改设备参数 |
| 真实 payload 确认 | 否 | 需要客户提供样例 |
| 真实控制命令确认 | 否 | 需要客户提供断开/恢复命令格式 |
| 现场联调 | 否 | 需要服务器、Broker、域名、证书准备完毕 |

## 7. 推荐部署顺序

| 顺序 | 动作 | 责任方 |
| --- | --- | --- |
| 1 | 准备公网服务器 | 项目方 |
| 2 | 部署 MQTT Broker（推荐 EMQX） | 项目方 |
| 3 | 配置 `mqtt.usr-iot.endpage.net` 和证书 | 项目方 |
| 4 | 部署后端和 Web | 项目方 |
| 5 | 填写服务器上的 `backend/.env` | 项目方 |
| 6 | 后端连通 Broker，自检通过 | 项目方 |
| 7 | 客户现场修改真机 MQTT 参数 | 客户 / 硬件方 |
| 8 | 采集真实 `status / alarm / feedback` 样例 | 客户 / 项目方 |
| 9 | 根据真实报文做最后一轮字段适配 | 项目方 |
| 10 | 做整套真机联调和交付验收 | 双方 |

## 8. 一句话结论

当前后端已经基本准备好。  
真正阻塞真机联调的，不是后端代码，而是：

- 服务器未部署
- MQTT Broker 未部署
- TLS 证书未准备
- 真机未切到你们自己的 Broker
- 真实报文和控制命令还没最终确认
