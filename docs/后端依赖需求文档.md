# 4G 压力报警联动控制系统后端依赖需求文档

## 一、文档目的

本文档用于明确后端首期开发所需的基础依赖、用途说明、安装边界与后续扩展建议，作为 `backend/requirements.txt` 的说明文档。

## 二、依赖选型原则

结合项目现有文档，后端依赖选型遵循以下原则：

1. 优先满足当前需求文档中的既定技术栈。
2. 优先选择稳定、成熟、维护成本低的依赖。
3. 首期以“快速搭建、可运行、可验证”为目标，避免过早引入复杂基础设施。
4. 对设备通信、鉴权、数据库、任务调度等关键模块优先使用主流方案。

## 三、首期必需依赖

### 3.1 Web API 与服务运行

| 依赖 | 是否必需 | 用途 |
| ---- | ---- | ---- |
| FastAPI | 必需 | 构建后端 API 服务 |
| Uvicorn | 必需 | 运行 FastAPI 服务 |

### 3.2 数据库与配置

| 依赖 | 是否必需 | 用途 |
| ---- | ---- | ---- |
| SQLAlchemy | 必需 | ORM 与数据库访问 |
| aiosqlite | 必需 | SQLite 异步驱动 |
| Pydantic | 必需 | 数据校验与模型定义 |
| pydantic-settings | 必需 | 环境变量与配置管理 |

### 3.3 登录鉴权

| 依赖 | 是否必需 | 用途 |
| ---- | ---- | ---- |
| python-jose[cryptography] | 必需 | JWT 生成与校验 |
| passlib[bcrypt] | 必需 | 密码哈希与校验 |
| python-multipart | 必需 | 表单提交与登录接口兼容 |

### 3.4 设备通信与调度

| 依赖 | 是否必需 | 用途 |
| ---- | ---- | ---- |
| paho-mqtt | 必需 | 设备 MQTT 通信接入 |
| APScheduler | 必需 | 心跳检测、离线检查、补发任务 |

```text

```

## 五、当前不建议首期引入的依赖

为控制复杂度，当前不建议在第一阶段引入以下能力：

| 类型 | 暂不引入原因 |
| ---- | ---- |
| Redis | 当前项目还未进入高并发和复杂缓存阶段 |
| Celery / Dramatiq | 当前定时与补发任务可先由 APScheduler 承担 |
| Alembic | 首期可以先快速搭建模型，待表结构稳定后再补数据库迁移 |
| PostgreSQL / MySQL 驱动 | 当前文档默认 SQLite，先保证开发效率 |
| Prometheus / OpenTelemetry | 可在后端主流程稳定后再增加监控能力 |

## 六、依赖安装建议

建议在 `backend/` 目录下使用独立虚拟环境，并通过以下命令安装：

```bash
pip install -r requirements.txt
```

如果后续增加测试依赖，可单独拆分为：

- `requirements.txt`：运行时依赖
- `requirements-dev.txt`：开发与测试依赖

## 七、后续建议补充的开发依赖

后端进入正式开发后，建议补充以下开发依赖：

| 依赖 | 建议阶段 | 用途 |
| ---- | ---- | ---- |
| pytest | 第二阶段 | 单元测试与接口测试 |
| pytest-asyncio | 第二阶段 | 异步测试支持 |
| httpx | 第二阶段 | API 测试客户端 |
| ruff | 第二阶段 | 代码检查 |
| black 或 ruff format | 第二阶段 | 统一格式化 |
| alembic | 第三阶段 | 数据库迁移管理 |

## 八、与项目需求的对应关系

依赖与需求映射如下：

| 需求能力 | 主要依赖 |
| ---- | ---- |
| 登录与鉴权 | FastAPI、python-jose、passlib |
| 设备状态查询 | FastAPI、SQLAlchemy、Pydantic |
| 报警记录查询 | FastAPI、SQLAlchemy |
| 手动控制 | FastAPI、paho-mqtt |
| 报警联动 | paho-mqtt、APScheduler、SQLAlchemy |
| 在线离线检测 | APScheduler、SQLAlchemy |

## 九、结论

当前后端首期依赖已经足够支撑以下基础建设：

1. 初始化 FastAPI 服务骨架。
2. 建立 SQLite 异步数据库访问能力。
3. 预留 JWT 鉴权能力。
4. 为 MQTT 设备接入与定时任务提供基础依赖。

后续建议先完成业务骨架与核心 API，再根据测试、迁移、监控等实际需要逐步增加开发依赖。
